### SEC-01 [CRIT] Webhook signature verification is structurally broken
**File:** `service/ClockifyWebhookService.java` — `verifyStoredWebhookToken()`  
**Problem:** The method compares `signature` (the full incoming Clockify-Signature JWT) against the stored `authToken` (the shared secret used to register the webhook) via `constantTimeEquals`. These are structurally different values and will never match. Every legitimate webhook is rejected with 403, or if the caller catches the exception, the signature check is effectively bypassed.  
**Fix:**
```java
// Use the SDK's signature parser with the stored shared secret:
boolean valid = ClockifySignatureParser.verify(signature, stored.authToken());
if (!valid) throw new ClockifyAccessForbiddenException("Webhook auth token mismatch");
```
Verify the exact SDK API in `../../ClockifyAddonAIPack/03-ai-pack/starter-java-sdk/`.

### BUG-01 [CRIT] `processDueJobs` — single `@Transactional` across all jobs; exception on job N rolls back DB deletes for jobs 0..N-1
**File:** `service/EstimateGuardService.java` — `processDueJobs()`  
**Problem:** The `@Transactional` wraps the entire loop. If job N throws (e.g. `stopRunningTimer` fails), all `deleteByJobId` rows for already-fired jobs 0..N-1 are rolled back. Those jobs are picked up again next tick, re-stopping already-stopped timers.  
**Fix:** Remove `@Transactional` from `processDueJobs`. For each job, use a `REQUIRES_NEW` helper that claims (deletes) the job and commits before the Clockify HTTP call:
```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
boolean claimJob(String jobId) {
    return cutoffJobStore.deleteByJobId(jobId) > 0;
}
// Then HTTP calls run outside any transaction.
```

### BUG-02 [CRIT] `Instant.toString()` sends nanosecond precision; Clockify stop-timer expects millisecond format
**File:** `service/EstimateGuardService.java` lines 228, 521  
**Problem:** `job.cutoffAt().toString()` emits `2026-04-19T14:23:01.123456789Z`. Clockify's stop-timer endpoint expects `yyyy-MM-dd'T'HH:mm:ss.SSS'Z'`. Nanosecond precision causes 400 errors, leaving timers running past cap.  
**Fix:**
```java
private static final DateTimeFormatter CLOCKIFY_TIMESTAMP =

// Usage:
backendApiClient.stopRunningTimer(installation, job.userId(), CLOCKIFY_TIMESTAMP.format(job.cutoffAt()));
```

### BUG-03 [HIGH] `RateInfo.present()` treats explicitly-set zero rate as absent — zero-rate users trigger fail-closed lock

Update JSON deserialization in `ClockifyJson` to set `configured=true` when the `amount` field is present in JSON.
